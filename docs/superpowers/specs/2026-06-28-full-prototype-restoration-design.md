# 全量原型还原设计方案

## 目标

按照 32 个页面的 HTML 原型和项目设计规范，补齐当前 Taro 微信小程序的业务功能。优先还原真实业务闭环，而不是单纯追求静态页面数量。最终目标是形成可用的产品路径：入口分发、用户动作、状态反馈、用户记录和后台处理能够串起来。

## 项目背景

本项目是 Taro 4 + React 18 + NutUI React Taro + Tailwind CSS v3 + Webpack5 的微信小程序。行商书苑面向财税机构、本地服务商和平台运营方，承载资源采购、线下活动、商业撮合、会员权益和运营管理等业务。

当前代码已经具备：

- 主包 tab 页：首页、服务商城、行商书苑、我的。
- 分包页面：资源、活动、会员、商机、用户中心、后台管理。
- 路由工具：`src/shared/router.ts`。
- 请求与业务接口模块：`src/shared/request.ts`、`src/services/`。
- 已有业务组件：`SectionCard`、`ItemList`、`ActionBar`、`StatGrid`、`FieldList`、`FormFields`、`FormPreview`、`EmptyState`。

参考资料：

- 产品原型：`docs/reference/001-requirements.html`
- 设计规范：`docs/reference/002-design-spec.html`
- 视觉规范：`design-system/行商书苑/MASTER.md`
- 架构说明：`docs/architecture.md`

## 已确认决策

### 范围

本次只做微信小程序端。H5 适配和 H5 构建验证不在范围内，除非后续明确要求。

原型还原按业务域和业务闭环推进，不按 1-32 页机械顺序实现。

### 数据策略

页面只展示真实接口返回的数据。

- 不硬编码假的业务数据来模拟订单数量、权益、营收、成交额、用户记录或后台记录。
- 核心字段缺失时，展示明确空态或业务安全占位，例如“暂无可展示的会员权益”“该活动暂未配置核销信息”。
- 非核心字段缺失时，模块可以隐藏，但缺失字段必须进入字段缺口清单。
- 列表页在真实数据足够时保留结构完整性，可以显示精简卡片。
- 详情页不为了贴近原型密度而填充假内容。
- 后台页不能伪造审核、订单、核销、需求或撮合记录。

每个页面还原时需要记录：

- 原型字段或模块。
- 使用的接口字段。
- 字段存在时的展示策略。
- 字段缺失时的处理方式。
- 缺口类型：`暂不展示`、`接口待补`、`后续版本`。

### 组件策略

优先使用 NutUI React Taro 和项目已有业务组件。组件库或已有组件能覆盖的场景，不重复手写等价控件。

只在多个页面反复出现同类结构时，新增轻量业务组件：

- `BusinessPageHeader`：通用页面标题、说明和主操作。
- `BusinessTabs`：状态/分类筛选。
- `DetailSection`：详情页分组信息。
- `StatusTimeline`：订单、报名、审核、支付、撮合等状态时间线。
- `AdminReviewCard`：后台审核、确认、撮合工作项。
- `FormSubmitBar`：表单底部主次操作栏。

这些组件应是 NutUI、Tailwind token 和现有页面规范上的薄封装。第一轮先完成业务行为和状态覆盖。后续 UI 改版时，优先替换这些组件内部实现，而不是逐页重写。

### 视觉标准

遵循 `design-system/行商书苑/MASTER.md`。

- 产品定位是专业的财税商业服务生态，不是阅读、传统文化或内容社区产品。
- 深蓝和白色作为主骨架。
- 金色只用于高价值动作、会员权益、活动报名和关键数据。
- 正式 UI 图标不使用 emoji，优先使用 `AppIcon`、NutUI 图标、SVG 或本地资产。
- 卡片圆角和间距保持克制。
- 当前阶段不要求像素级一致，但信息层级、主操作和状态表达必须符合原型。

## 阶段计划

### 阶段 0：导航、角色、字段覆盖和状态规范

目的：先明确边界，再补页面。

交付内容：

- 审计所有 tab 页和分包页面路由。
- 梳理普通用户、会员用户、认证机构、服务商、后台运营/审核人员的角色与权限矩阵。
- 建立 32 个原型页面的字段覆盖表。
- 统一加载、空态、错误、无权限、认证中、审核中、已失效、已取消、已完成等状态规则。
- 确认第一批轻量业务组件的边界和归属。

验收标准：

- 现有路由能映射到对应原型页面。
- 新增导航不散落手写路由字符串。
- 缺失接口字段能在缺口清单中看到。
- 空态和权限态文案一致。

### 阶段 1：首页/服务商城/书苑入口与核心列表

目的：先建立业务入口分发和核心发现路径。

页面与区域：

- 首页 tab。
- 服务商城 tab。
- 行商书苑 tab。
- 资源首页和资源列表。
- 活动首页和活动列表。
- 商机首页和商机列表区域。

交付内容：

- 入口卡片和跳转路径对齐原型业务模块。
- 资源、活动、商机核心列表由真实接口驱动。
- 分类/状态筛选使用接口返回的真实分类，或安全默认值“全部”。
- 列表页具备空态和错误态。
- 记录入口页和列表模块的字段覆盖情况。

验收标准：

- 用户能从一级 tab 进入资源、活动和商机流程。
- 核心列表能展示真实接口数据，或明确解释为什么暂无数据。
- 不展示假的统计数字或仅存在于原型中的业务数字。

### 阶段 2：资源、活动、商机转化闭环

目的：补齐用户侧核心业务动作。

页面与区域：

- 标品资源详情。
- 非标资源详情。
- 资源需求提交。
- 资源采购确认。
- 活动详情。
- 活动报名。
- 活动拼团参与。
- 电子票/报名成功。
- 商机详情。
- 发布商机。
- 申请接单。

交付内容：

- 详情分组展示真实接口字段。
- 表单使用 NutUI 和共享字段组件实现。
- 有可用接口时，提交动作接入现有 service 方法。
- 提交、报名、采购、发布、申请后有明确状态反馈。
- 接口支持时，提供用户记录入口。

验收标准：

- 资源流程支持查看详情，并基于真实数据和可用接口进入采购或提交需求。
- 活动流程支持详情、报名意向和成功/电子票状态。
- 商机流程支持详情、发布/申请表单和真实接口状态反馈。
- 缺字段或缺接口进入缺口清单，不用假数据补齐。

### 阶段 3：会员、我的、订单、权益、积分、评价

目的：补齐留存、会员和个人记录链路。

页面与区域：

- 会员权益页。
- 会员开通确认。
- 对公支付凭证。
- 我的 tab。
- 企业认证页。
- 我的订单。
- 我的活动。
- 我的权益。
- 我的积分。
- 我的评价。
- 用户设置。

交付内容：

- 会员权益和确认页由真实 VIP/config 接口驱动。
- 支付凭证流程只使用真实支付/订单状态接口。
- 我的页菜单和指标对齐真实用户、学习、订单、消息、认证数据。
- 用户记录列表具备加载、空态、错误和状态展示。
- 清晰展示认证状态和会员状态。

验收标准：

- 会员、订单、权益、积分、评价页面能说明数据来源和状态含义。
- 不展示假的会员权益、积分余额、订单金额或评价记录。
- 依赖认证的业务动作有清晰状态门槛。

### 阶段 4：后台运营闭环

目的：让后台页面承接前台动作或接口真实工作项。

页面与区域：

- 后台核销页。
- 后台认证审核。
- 后台订单确认。
- 后台资源需求。
- 后台商机撮合。

交付内容：

- 后台工作项卡片使用真实记录。
- 审核、确认、撮合等动作接入可用接口。
- 每类工作项展示状态时间线或字段预览。
- 后台页具备空态和无权限态。

验收标准：

- 后台页面对应真实前台动作或真实后端记录。
- 核销、审核、订单、需求、撮合状态明确。
- 不为了页面密度伪造后台工作项。

### 阶段 5：视觉一致性和组件收口

目的：功能还原后减少 UI 漂移。

交付内容：

- 把重复页面模式收敛到轻量业务组件。
- 按设计规范统一主要页面的信息层级、间距、卡片样式、图标、标签、按钮和空态。
- 复核 32 个原型页面，并标记还原状态。
- 删除已被共享组件覆盖的局部一次性 UI。

验收标准：

- 32 个原型页面都有明确状态：`已实现`、`部分实现`、`接口缺失`、`暂缓`、`不在范围内`。
- 关键页面的信息架构和主操作符合原型。
- 后续 UI 改版可以通过替换组件内部实现推进。

## 核心业务闭环

实现时要验证业务闭环，而不是把页面当成孤立屏幕：

- 资源/服务：浏览资源或服务 -> 查看详情 -> 采购、咨询或提交需求 -> 查看状态或记录。
- 活动：浏览活动 -> 查看详情 -> 报名 -> 查看报名结果或电子票 -> 接口支持时可被后台核销引用。
- 商机：浏览或发布商机 -> 查看/申请 -> 状态反馈 -> 用户记录或后台撮合。
- 会员：查看权益 -> 确认开通/升级 -> 支付或凭证状态 -> 我的页会员状态。
- 认证：提交认证 -> 审核状态 -> 后台审核 -> 用户侧认证状态。
- 后台：接收前台或后端记录 -> 审核、确认、核销或撮合 -> 接口支持时反映状态变化。

## MVP 验收标准

- `pnpm typecheck` 通过。
- 阶段涉及路由配置、应用配置、构建配置、全局样式，或准备阶段验收时，`pnpm build:weapp` 通过。
- 4 个 tab 页和所有分包页面都能通过路由配置进入。
- 新增导航使用 `routes` 和 `router`，不散落手写路径。
- 每个业务域在接口支持范围内具备列表、详情或记录、主操作、反馈状态、用户或后台后续处理。
- 接口数据按真实返回展示，不引入假的业务数据。
- 必要页面状态覆盖加载、空态、错误、无权限、审核中、已失效/已取消/已完成等场景。
- 资源、活动、商机各至少有一个可验证的用户侧闭环。
- 会员、订单、积分、权益、评价页面能解释数据来源和状态含义。
- 后台页面处理真实记录，或展示清晰空态/权限态。
- 字段覆盖表能识别被接口缺失阻塞的原型模块。
- 视觉层级、主操作和组件使用符合设计规范。

## 不在范围内

- H5 适配。
- 像素级视觉重构。
- 面向生产页面的假 mock service 层。
- 大规模架构重写。
- 没有明确业务收益的复杂状态机。
- 后端 API 设计或实现。

## 主要风险

- 真实接口可能无法覆盖所有原型模块，页面看起来会比 HTML 原型更“空”。
- 非核心模块隐藏后，如果不维护缺口清单，接口缺口容易被忽略。
- 如果当前环境中前台动作不会生成对应后端记录，后台页面验收会受限。
- 支付和会员流程如果当前 API 只支持转账凭证或状态轮询，需要额外明确产品边界。
