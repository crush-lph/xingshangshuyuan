# 业务闭环任务清单

本文档记录基于当前原型、现有页面实现和现有接口能力后的业务闭环缺口。当前接口以 `docs/api.md`、`docs/默认模块.openapi.json` 和 `src/services/` 已生成 service 为准；缺少接口时不在前端制造假状态或假处理结果，而是明确反馈为接口待补。

## 闭环判断口径

- `页面覆盖`：用户能看到对应页面和核心信息。
- `发起动作`：用户能提交表单、创建订单、报名、申请或上传材料。
- `后台处理`：运营端能审核、确认、核销、撮合或变更状态。
- `状态流转`：前台和后台能看到同一业务记录在不同阶段的真实状态。
- `结果反馈`：用户能看到通过、驳回、到账、开通、核销、撮合成功或失败原因。

当前项目页面覆盖较高，前台发起动作已覆盖一部分；剩余工作主要集中在后台处理、状态流转和结果反馈。

## 缺少接口反馈清单

### 支付与订单

- 后台待确认订单列表：支持按状态筛选对公转账、会员开通、资源采购订单。
- 后台确认到账接口：输入 `order_no`、处理备注，返回最新订单状态。
- 后台驳回凭证接口：输入 `order_no`、驳回原因，返回最新订单状态。
- 订单状态枚举说明：至少区分待支付、待上传凭证、待财务确认、已支付/已开通、已驳回、已取消、已完成。
- 凭证专用上传/绑定接口：当前 `uploadFileRecord` 和 `paymentCallback` 能承接基础动作，但缺少明确的对公凭证业务结果。

### 会员权益

- 会员开通确认结果接口：支付确认后返回当前 `vip_level`、有效期和权益生效状态。
- 用户权益使用明细接口：返回权益项、总额度、已使用、剩余、有效期、使用记录。
- 会员订单与权益生效状态关联字段：用于用户侧明确展示“待确认/已生效/开通失败”。

### 企业认证

- 后台认证申请列表接口：支持待审核、已通过、已驳回筛选。
- 后台认证详情接口：返回企业资料、证照附件、提交人、提交时间、历史审核记录。
- 后台认证通过接口。
- 后台认证驳回接口：需要驳回原因。
- 用户侧认证状态枚举说明：未提交、审核中、已通过、已驳回、需补充材料。

### 商机撮合

- 后台商机审核列表接口：支持待审核、进行中、已关闭、已撮合状态。
- 商机申请列表接口：按 `opportunity_id` 返回申请方、报价、附件、状态。
- 后台选择/推荐服务商接口。
- 后台撮合成功/失败接口：记录处理人、处理备注、结果状态。
- 用户侧申请状态接口：申请人可看到待审核、已入围、未入围、已撮合等状态。

### 活动与票务

- 活动报名支付状态字段：明确免费、待支付、待确认、已报名、已取消、已核销。
- 付费报名支付确认后生成电子票的结果字段。
- 拼团详情接口：成团人数、当前成员、截止时间、拼团状态。
- 拼团参团/取消/支付结果接口。

### 资源需求与交付

- 资源需求列表接口：区分用户提交记录和后台待处理记录。
- 后台资源需求分配接口：指定供应商、客户经理或处理人。
- 资源需求状态更新接口：待处理、处理中、已匹配、已关闭、已完成。
- 资源采购交付进度接口：服务开通、交付中、已交付、售后中。

## 实施建议

1. 前端先保持当前接口边界，不做假审核、假到账、假撮合。
2. 后台页面优先展示真实数据和明确缺口说明，避免给运营造成“已能处理”的误导。
3. 新接口补齐后，优先从 P0 闭环接入：支付确认、会员生效、认证审核、订单确认、商机撮合。
4. 每补一个接口，回写 `docs/product/field-coverage.md` 和 `docs/product/prototype-restoration-map.md`，同步页面状态。
